# 当前预测实现审计

## 结论摘要

当前项目中的“预测”功能并未沉到共享核心层，而是桌面端和移动端各维护一套本地实现。两端输入数据口径基本一致，但算法细节和输出行为已经出现明显分叉。现阶段最合理的后续方向是：

1. 先把预测算法抽成 `packages/core` 中的纯函数模块。
2. 再在共享算法之上迭代更稳健的预测策略，而不是继续分别修 Electron / Expo 两套实现。

## 代码位置

### 桌面端

* 视图入口：`src/renderer/views/Prediction.tsx`
* 算法实现：`src/renderer/services/PredictionService.ts`

### 移动端

* 页面入口兼算法实现：`apps/mobile/app/(tabs)/prediction.tsx`

### 共享层

* `packages/shared` 只提供 `IRecord` 等类型。
* `packages/core` 目前没有 prediction 相关导出。

## 数据流与输入口径

两端预测都基于未删除记录：

* 桌面端 `src/main/database.ts -> DatabaseService.GetRecords()` 查询 `Deleted = 0`
* 移动端 `apps/mobile/src/services/MobileDatabaseService.ts -> GetRecords()` 查询 `Deleted = 0`

两端都在各自数据服务层把时间字段从 ISO string 转成 `Date`，所以预测层输入口径基本一致：

* `StartTime: Date`
* `EndTime: Date`
* `Duration: number`

## 两套算法的真实差异

### 桌面端算法

文件：`src/renderer/services/PredictionService.ts`

行为：

* 先按 `EndTime` 升序排序。
* `DaysSinceLast`：`now - last.EndTime`，单位天。
* `AverageInterval`：只取最近最多 30 条记录，计算相邻记录间平均间隔，单位天。
* `PeakHour`：统计全部记录 `EndTime.getHours()` 的众数。
* `Level`：根据 `daysSinceLast / averageInterval` 分为 `low / medium / high / veryHigh`。
* `NextEstimate`：`last.EndTime + averageInterval`，然后把小时强制改为 `PeakHour`，若结果已过去则每天顺延直到未来。

### 移动端算法

文件：`apps/mobile/app/(tabs)/prediction.tsx`

行为：

* 先按 `EndTime` 升序排序。
* `averageGapMinutes`：对全量历史记录求相邻记录平均间隔，单位分钟。
* `nextEstimate`：`last.EndTime + averageGapMinutes`
* `daysSinceLast`：`now - last.EndTime`，单位天。
* 文案层只有“样本不足 / 趋势稳定”两态，没有等级、活跃时段、未来滚动修正。

## 发现的问题

### 1. 共享边界失效

文档把预测算法定义为可复用纯逻辑，但实现没有进入 `packages/core`。这会带来：

* 同一业务含义有两套代码
* 修 bug 需要双端同步修改
* 任何算法升级都要双写和双测

### 2. 两端行为不一致

几个关键不一致点：

* 桌面端用最近 30 条；移动端用全量历史。
* 桌面端把预测时间强行校正到未来；移动端可能给出已经过去的预测时间。
* 桌面端输出等级和高峰时段；移动端没有。
* 移动端“趋势稳定”实际上只是“有至少一个间隔可算”，表达过强。

### 3. 简单平均对异常样本敏感

当前核心估计都基于“相邻记录时间差的算术平均”。这会受到以下场景干扰：

* 长时间停用后回归
* 假期或特殊时期的短时间高频记录
* 很早以前的使用习惯与最近习惯不一致

### 4. 高峰时段统计过于粗糙

桌面端 `PeakHour` 直接统计所有记录小时众数，问题包括：

* 全量历史会掩盖近期作息变化
* 并列峰值没有 tie-break 语义
* 小时粒度过粗，`xx:59` 和 `xx:01` 被分到相邻桶

### 5. 缺少样本质量和不确定性表达

当前没有这些信息：

* 最小样本门槛
* 波动程度（例如标准差、分位区间）
* 预测可信度
* “无法稳定预测”的明确判断

结果是 UI 会把本质很弱的估计也包装成确定性结论。

### 6. 没有预测测试

未发现 prediction 相关单元测试。后续一旦改算法，如果没有固定样例，很难确保桌面端和移动端在边界条件下保持一致。

## 推荐的后续改造顺序

### 方案 1：先统一，再升级算法（推荐）

步骤：

1. 在 `packages/core` 新增纯函数 prediction 模块。
2. 把排序、间隔计算、样本门槛、输出结构统一。
3. 桌面端和移动端都只消费共享结果，再保留各自 UI 层格式化差异。
4. 补上纯函数测试后，再迭代算法。

优点：

* 先解决结构性重复
* 后续改算法只动一处
* 风险可控，容易回归验证

缺点：

* 第一阶段收益主要在结构清理，用户感知不一定立刻明显

### 方案 2：先重做算法，再整理复用

步骤：

1. 直接设计新版算法
2. 分别接入桌面端和移动端
3. 最后再尝试抽共享层

优点：

* 用户侧效果改善更快

缺点：

* 会在两套实现里重复搬运复杂逻辑
* 结构债会进一步增加

不推荐。

## 推荐的算法改进方向

不直接给“机器学习化”建议，先从可解释、低风险的规则模型开始。

### 第一层：统一基础统计

建议输出统一结构，例如：

* `sampleSize`
* `recentSampleSize`
* `daysSinceLast`
* `medianIntervalDays`
* `meanIntervalDays`
* `intervalStdDevDays`
* `predictedWindowStart`
* `predictedWindowEnd`
* `confidence`
* `peakHours`
* `status`

### 第二层：用稳健统计替换简单平均

建议优先考虑：

* 最近 N 条样本窗口
* 中位数间隔 `median`
* 截尾平均 / winsorized mean
* 用 IQR 或标准差描述波动

理由：

* 比纯平均更抗离群点
* 容易解释
* 纯函数实现简单，测试成本低

### 第三层：从“时点预测”改成“时间窗预测”

不要只返回一个确定时刻，改为：

* 最早可能时间
* 最可能中心时间
* 最晚可能时间

这会比“下一次一定在某个时刻左右”更诚实。

### 第四层：加入样本门槛和降级策略

例如：

* `< 2` 条记录：不给预测
* `2 ~ 4` 条记录：只显示平均间隔，不显示置信结论
* 样本波动过大：显示“模式不稳定”

### 第五层：近期与历史分层

高峰时段和间隔都更应该偏向近期记录。可考虑：

* 间隔只看最近 N 条
* 高峰时段使用最近 N 条或衰减权重
* 当近期与历史差异明显时，优先近期

## 我的建议

进入实现时，优先做下面这一条：

**先把 prediction 抽到 `packages/core`，并用稳健统计重写为“样本门槛 + 中位数/截尾平均 + 预测时间窗 + 置信度”的共享纯函数。**

这是当前投入产出比最高的方向。

## 讨论后收敛的统一算法草案

根据本轮讨论，统一算法建议收敛为下面这版：

### 1. 核心目标

* 小样本下可工作，但不强行给假精确预测
* 优先输出预测窗口，而不是单一时点
* 当窗口过宽时降级，而不是无限压低置信度
* 复杂度限制在独立 prediction 模块内部

### 2. 输入与样本

* 使用记录的 `EndTime`
* 按时间排序后计算相邻间隔
* 仅使用最近 `N=10` 个间隔参与预测；样本不足时使用全部可用间隔
* 这里的 `N` 指相邻间隔数，不是记录条数；`10` 个间隔大约对应 `11` 次记录
* 当真实间隔样本数 `< 2` 时，不输出预测窗口

### 3. 弱先验

* 先验中心：`3 天`
* 先验强度：`k = 1`
* 含义：相当于 1 个虚拟间隔样本
* 随真实样本增加，先验权重自动衰减

建议实现方式使用简单收缩估计，而不是复杂贝叶斯采样。

### 4. 中心估计

* 对近期相邻间隔做稳健中心估计
* 可选实现优先级：
  1. 收缩均值
  2. 收缩中位数
  3. 截尾均值

本轮倾向先用**简单收缩均值 + 稳健波动估计**，保持实现可控。

### 5. 波动估计

* 使用 `MAD` 或 `IQR`
* 设定最小波动下限，避免区间假窄

### 6. 置信区间与产品阈值

* 依次尝试 `95% -> 90% -> 85%`
* 选择第一个满足 `halfWidth <= 1.5 天` 的区间
* `85%` 仍不满足时，不展示精确预测窗口

### 7. 降级策略

* `n < 2`：显示“样本不足，继续记录后再预测”
* `n >= 2` 但窗口过宽：显示更粗粒度提示，而不是伪精确时间

### 8. 时间展示

时间桶只用于展示，不进入预测模型：

* `00:00-06:00`
* `06:00-12:00`
* `12:00-18:00`
* `18:00-24:00`

内部预测继续使用连续时间，最终展示时再映射到时间桶。
