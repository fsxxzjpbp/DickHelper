# Data Migration — Old Version Import Contract

> How the new version imports data exported from the old DickHelper.

---

## Source

Old version's **"导出数据"** button in `RecordForm.tsx` (line 128-139):

```typescript
const handleExport = () => {
    const data = StorageService.exportData();  // JSON.stringify(records, null, 2)
    const blob = new Blob([data], { type: 'application/json' });
    // ... triggers download as "masturbation_records.json"
};
```

The exported file is `masturbation_records.json` — a JSON array of all records.

---

## Exact JSON Format

```json
[
  {
    "id": "b3f1a2c4-...",                      // UUID v4 string
    "startTime": "2025-02-21T14:30:00.000Z",    // ISO 8601 — BUT actually stores END time (see field semantics below)
    "duration": 5.5,                             // minutes, up to 2 decimal places
    "notes": "optional text"                     // string, field absent if no notes
  }
]
```

### Field Contract

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | `string` (UUID v4) | Yes | Generated by `uuid` npm package |
| `startTime` | `string` (ISO 8601) | Yes | **Misnamed**: actually the end time when user clicked "结束" |
| `duration` | `number` | Yes | Minutes, precision via `.toFixed(2)` then `Number()` |
| `notes` | `string \| undefined` | No | Absent from JSON if empty/undefined |

### Field Semantics — startTime Bug

Despite the name `startTime`, this value is set in `RecordForm.tsx:73`:

```typescript
const endTime = new Date();
// ...
const record = {
    id: uuidv4(),
    startTime: endTime,  // ← BUG: stores endTime, not startTime
    duration: ...,
};
```

**Resolution for new version**: During import, interpret `startTime` as the **record completion time** (end time). The new data model should use `endTime` as the canonical field name and map old `startTime` → `endTime` during import.

The old version's import validation in `StorageService.importData()`:

```typescript
// Checks performed:
typeof record.id === 'string'       // UUID
record.startTime                     // truthy check only (any non-empty value passes)

// NOT checked:
//   - notes type
//   - id format (any string passes)
//   - startTime is valid date string
//   - duration range sanity
//   - array items are objects
```

## New Version Import Strategy

### Phase 1: Detect old format

Heuristic: if the JSON file is an array and each item has `id`, `startTime`, `duration` (but no `version` field), treat as old DickHelper export.

### Phase 2: Validate & Transform

```
For each record in array:
  1. Validate: id (string), startTime (parseable ISO), duration (number >= 0)
  2. Skip invalid records, count failures
  3. Map: old.startTime → new.endTime
  4. Keep: id, duration, notes (sanitize: toString() if not string/undefined)
  5. Add: importedAt = now, sourceVersion = "legacy-dickhelper-v0"
```

### Phase 3: Merge Strategy

- Default: **insert all** (not replace — user may already have new records)
- Dedup by `id`: skip records whose `id` already exists in the database
- Report: "Imported X records, skipped Y duplicates, rejected Z invalid"

### Phase 4: Schema Version

New format includes a `version` field so future migrations can be deterministic:

```json
{
  "version": 1,
  "records": [...]
}
```

### Soft Delete Fields (v2+)

Records may now include `Deleted` and `DeletedAt` fields for tombstone support:

```json
{
  "id": "b3f1a2c4-...",
  "startTime": "2025-02-21T14:30:00.000Z",
  "duration": 5.5,
  "notes": "optional text",
  "deleted": 0,
  "deletedAt": null
}
```

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `deleted` | `number` (0 or 1) | No | Defaults to 0 if absent (backward compatible) |
| `deletedAt` | `string \| null` | No | ISO 8601 UTC, null if not deleted |

**Import behavior**: Records with `deleted=1` are imported as tombstones (not shown in main list). This enables sync to propagate deletions across devices.
